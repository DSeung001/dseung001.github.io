프로젝트

Lecture Streaming Service(개인) 2026.05 - 현재

학원 수강생이 강의 영상을 업로드하고 시청할 수 있는 영상 관리 서비스를 개발하고 있습니다.

이 프로젝트는 동영상 도메인으로 커리어를 확장하기 위해 직접 설계하고 구현한 프로젝트입니다. 단순히 영상을 업로드하는 기능이 아니라, 업로드된 영상을 비동기로 처리하고 HLS로 변환한 뒤 S3와 CloudFront를 통해 제공하는 구조입니다.

주요 기능

  • User: 회원가입 및 로그인 / 업로더 권한 승인
  • Media: 커리큘럼 관리 / 강좌 관리 / 강의 타임라인 관리 / 영상 업로드 / HLS 인코딩 결과 배포
  • Job: 영상 시청 플로우 / 영상 Publish 작업 관리
  • QnA: 자유 QnA / 마크다운 입력 지원 (QnA/댓글)

기술 구조

Client → Backend API → S3 Staging Upload → PublishJob 생성 
→ Celery / Redis 비동기 작업 → FFmpeg HLS 인코딩 → S3 저장
→ CloudFront 배포 → Client 재생

문제 및 개선

이슈 해결
긴 영상 인코딩으로 업로드 API·사용자가 동기 대기 PublishJob 생성 후 Celery 비동기 인코딩, 202 응답과 진행률 폴링 UI로 분리
8GB EBS에서 docker compose build 시 이미지·캐시가 겹치며 디스크 부족 Next.js Standalone, 백엔드/Celery 이미지 통합, Docker 로그 제한으로 약 600MB 절감
워커가 staging 영상 전체를 tmp에 받아 원본·HLS가 동시에 디스크를 점유 presigned URL로 FFmpeg가 S3 원본을 HTTP 스트리밍 입력, HLS 산출물만 로컬에 보관
Redis visibility_timeout(1h)·prefetch(4) 조합으로 완료 Job이 재배달되어 NoSuchKey 발생 visibility_timeout 4시간, prefetch 1, 완료 Job 재실행 방지 및 NoSuchKey 재시도 제외
task_time_limit 2시간으로 긴 영상 인코딩 중 워커가 끊기고 Job이 processing에 고정 task time limit을 4시간으로 확대
WebM/VFR 영상의 왜곡 FPS 메타데이터(1000 fps)로 인코딩이 비정상적으로 지연 FFmpeg 출력 FPS를 CFR 30으로 고정
강의 영상의 품질·용량·트래픽 균형 필요 CRF 24, maxrate 3M, bufsize 6M 적용 (VMAF 95+ 유지)
GB 단위 대용량 업로드로 서버 리소스 부담 Request size를 1GB로 제한 (backend env 설정)
